## 1. 概述

### 1.1 项目背景

小型团队（2-10人）在协作过程中面临三个核心问题：

1. **信息碎片化** — 文档在 Notion、讨论在 Slack、邮件在邮箱，关键信息散落各处
2. **知识流失** — 讨论结论随时间被遗忘，新成员难以获取上下文
3. **任务脱节** — 决策与执行分离，执行层不清楚决策背景

### 1.2 项目目标

打造一个具有持久记忆的 AI 助手，让 AI 成为团队的"老员工"。

### 1.3 核心价值

- **记忆能力**：长期持久存储团队知识，而非单次会话
- **关系推理**：不仅检索，还能推理实体间的关联
- **上下文理解**：理解决策背景，而非孤立的文档片段

## 2. 系统设计

### 2.1 整体架构

详见 [ARCHITECTURE.md](ARCHITECTURE.md)

### 2.2 多 Agent 协作

采用 4 个专业 Agent + 1 个 Orchestrator 的设计：

| Agent | 职责 | 输入 | 输出 |
|-------|------|------|------|
| Orchestrator | 任务编排 | 用户输入 | 任务分解 |
| Extract | 实体提取 | 原始文本 | 实体列表 |
| Comprehend | 语义理解 | 实体 + 文本 | 摘要 + 决策 |
| Relate | 关系关联 | 实体 + 理解结果 | 关系列表 |
| Store | 知识存储 | 所有结果 | 存储确认 |

### 2.3 知识库设计

**双层存储**：

1. **向量数据库（Qdrant）**
   - 语义检索
   - 文档分块
   - 全文索引

2. **图数据库（Neo4j）**
   - 实体节点
   - 关系边
   - 路径查询

**为什么需要两层？**

| 能力 | 向量检索 | 图检索 |
|------|---------|--------|
| 语义相似 | ✅ | ❌ |
| 关系推理 | ❌ | ✅ |
| 路径查询 | ❌ | ✅ |
| 模糊匹配 | ✅ | ❌ |

两者互补，实现 **RAG + Graph** 混合检索。

## 3. 数据模型

### 3.1 实体类型

| 类型 | 说明 | 示例 |
|------|------|------|
| Person | 团队成员 | 张三、李四 |
| Project | 项目 | 用户认证重构 |
| Decision | 决策 | 采用 JWT 方案 |
| Task | 任务 | 更新技术文档 |
| Document | 文档 | 技术设计文档 |
| Meeting | 会议 | 技术评审会议 |
| Message | 消息 | Slack 消息 |

### 3.2 关系类型

| 关系 | 说明 | 示例 |
|------|------|------|
| decided_by | 决策来源 | JWT方案 decided_by 技术评审 |
| affects | 决策影响 | JWT方案 affects 认证重构 |
| assigned_to | 任务分配 | 更新文档 assigned_to 张三 |
| part_of | 归属关系 | 任务 part_of 项目 |
| discussed_in | 讨论场合 | 方案 discussed_in 会议 |
| related_to | 一般关联 | A related_to B |

## 4. 核心流程

### 4.1 知识添加流程

```
数据源 → 解析 → 提取 → 理解 → 关联 → 存储
```

每一步都有明确的输出和验证，确保知识质量。

### 4.2 问答流程

```
用户提问 → Query 改写 → 混合检索 → 上下文组装 → LLM 生成 → 响应
```

### 4.3 决策追踪流程

```
会议内容 → 提取决策 → 关联影响 → 存入图谱 → 可视化链路
```

## 5. 技术选型

### 5.1 技术栈

| 组件 | 选型 | 理由 |
|------|------|------|
| LLM | Claude | 长上下文支持好 |
| 向量库 | Qdrant | 轻量、易部署 |
| 图库 | Neo4j | 成熟、社区活跃 |
| API | FastAPI | 异步、性能好 |
| 前端 | Streamlit | 快速原型 |

### 5.2 部署架构

**开发环境**：
- Docker Compose 启动本地服务
- Streamlit 前端
- 本地文件存储

**生产环境**：
- 云端 Qdrant + Neo4j
- Tauri 桌面客户端
- Cloudflare Tunnel 反向代理

## 7. 风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| LLM 幻觉 | 回答不准确 | RAG 约束、来源标注 |
| 数据一致性 | 图库/向量库不同步 | 异步写入、校验任务 |
| 隐私泄露 | 敏感信息外传 | 本地优先、脱敏处理 |
| 性能瓶颈 | 检索延迟高 | 缓存、分层索引 |
