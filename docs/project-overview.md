# TeamMind - 智能团队知识记忆系统

## 项目概述

### 一、核心痛点

在中小团队（2-10人）的日常协作中，我发现了三个普遍且严重的信息管理问题：

**痛点一：信息碎片化**。团队知识散落在 Slack 消息、邮件、会议纪要、文档、代码注释等多个渠道。当新成员加入或需要追溯历史决策时，往往要在 5-10 个平台间反复切换，还常常找不到关键信息。

**痛点二：知识断层与流失**。老员工离职带走隐性经验，项目重启时发现"当年为什么这么做"已成悬案。决策的背景和逻辑随时间蒸发，只剩下执行结果。

**痛点三：上下文割裂**。系统设计改了三版，但没人知道 v1 为什么被推翻。任务分配下去了，但不知道它源于哪个讨论、哪个决策。信息之间的关联断裂，导致重复讨论、决策反复。

这些问题不是工具能解决的——Notion 不会主动理解内容，Slack 搜索不到语义相似但措辞不同的讨论。我需要的是能够**理解**、**关联**、**记忆**团队知识的 AI 系统。

### 二、解决方案：TeamMind

TeamMind 是一个 AI Native 的团队知识记忆系统，让 AI 成为团队的"老员工"，持续学习和关联团队知识，解决信息碎片化与知识流失问题。

### 三、核心逻辑：多 Agent 协作 + 混合知识库

**架构设计**：采用 4 Agent 分工协作，避免单一大模型处理所有任务导致的效果不稳定。

```
用户输入（消息/文档/问答）
        ↓
Orchestrator Agent（意图识别）
    ├─ STORE 流程 → Extract → Comprehend → Relate → Store
    └─ QUERY 流程 → Query Rewrite → Hybrid Retrieval → Generate
```

**长链推理流程（STORE）**：

1. **Extract Agent**：从非结构化内容中提取实体（人名、决策、技术方案），识别信息类型（讨论/决策/任务）

2. **Comprehend Agent**：在提取实体基础上进行深度理解——生成摘要、识别决策点、提取行动项、归纳主题

3. **Relate Agent**：分析实体间关系（依赖、影响、对立），构建关联图谱，为每个实体建立上下文链路

4. **Store Agent**：双存储——向量化存 Qdrant 支持语义检索，关系数据存 Neo4j 支持图推理

**混合检索流程（QUERY）**：

1. Query Rewrite：改写问题，扩展同义词，提高检索召回率
2. Hybrid Retrieval：并行执行向量检索（语义相似）和图检索（关系链路）
3. Context Fusion：融合多跳结果，生成带引用的答案

### 四、技术栈

- **LLM**：Claude API（长上下文 + 工具调用）
- **向量数据库**：Qdrant（语义检索）
- **图数据库**：Neo4j（关系推理）
- **数据摄入**：n8n 自动化（Slack/Notion/邮件）
- **前端**：Streamlit Demo + React/Tauri 桌面端
- **部署**：本地优先 + Cloudflare Tunnel 云端访问

### 五、差异化优势

相比通用 RAG 方案，TeamMind 的核心创新在于：

1. **多 Agent 分工**：每个 Agent 专注单一任务，通过定义清晰的接口协作，降低单点失败风险，提高可维护性

2. **混合知识库**：向量检索找相似内容，图检索找关联链路，两者互补实现"不仅知道是什么，还知道为什么"

3. **本地优先**：核心数据（向量、图谱）本地存储，LLM 云端调用，兼顾数据隐私和 AI 能力

### 六、现状与计划

当前项目已具备可运行基线，包含：Agent 核心逻辑、Qdrant/Neo4j 适配层、Streamlit 演示界面、CLI 工具、FastAPI 最小入口和测试基线。

下一步计划：Slack/Notion 自动化摄入、多轮对话支持、权限模型和更完整的混合检索。
